# Overnight PSTACK run — plan + status

This file is the handoff for an overnight PSTACK iteration. Everything lives
under `evals/` so that the app, system prompt, and existing eval infra
(`eval_server/`) are untouched.

## What's running

The overnight queue (`evals/queue.json`) drives `evals/run_overnight.mjs`,
which for each entry:

1. Builds a hill batch via `evals/build_round.mjs` (5 variants × 10 runs = 50
   inferences targeting one concept).
2. Runs it through `eval_server/run_cactus.py` against the same INT4 weights
   the phone uses (Mac CPU path).
3. Scores with `evals/score.mjs` (glslangValidator compile pass-rate +
   shape checks).
4. Splits the raw by `variant` field and produces a per-variant breakdown.
5. Appends a summary row to `evals/rounds/overnight.jsonl`.

State is incrementally written. An interrupted run resumes by editing
`evals/queue.json` to skip what's done and restarting `run_overnight.mjs`.

## Variants per concept

Per PSTACK, every hill has the same 5 variants:

- **V0 baseline** — current `p0.txt` unchanged (control, required).
- **V1 rule** — `p0` + a one-line declarative rule for the concept.
- **V2 snippet** — `p0` + a 4–8 line GLSL snippet showing the pattern.
- **V3 reference** — `p0` + a complete working shader for the concept.
- **V4 constraint** — `p0` + a general "finish the shader, never stop
  mid-function" anti-truncation constraint (shared across concepts; tests
  whether the dominant failure mode is truncation rather than the actual
  effect).

Variant texts are baked into `evals/prompts/hill-<concept>.json` files,
generated by `evals/gen_variants.mjs`.

## What's *not* changing tonight

## Morning checklist (for me/you)

Run:

```
cd /Users/michael/github/voice-agents-hack
node evals/morning_report.mjs
```

It tabulates each hill: per-variant pass rate, best variant, delta over
baseline. Look for hills where best > baseline by **>+10 percentage points**
(≈ 1 std beyond single-batch noise at n=10).

For each clear win:

1. Re-run the hill once more to confirm (sampling variance is real).
2. If confirmed, append the winning variant's edit to `evals/prompts/p0.txt`
   to produce `p1.txt`.
3. Run a fresh full-batch validation (`evals/rounds/round-1-batch.json`)
   against `p1` to check for cross-concept regressions.
4. If clean, lock and continue with the next concept's hill against `p1`.

For null/regression results: mark the concept as ceiling-bound for this
generator and skip in future rounds.

## Files (reference)

- `PSTACK.md` — the algorithm (no project specifics).
- `evals/prompts/p0.txt` — seed system prompt (= current SHADER_SYSTEM_PROMPT).
- `evals/prompts/hill-<concept>.json` — per-concept variant specs.
- `evals/queue.json` — overnight execution order.
- `evals/rounds/round-0-*` — baseline full-batch run.
- `evals/rounds/hill-<concept>-r<N>-*` — per-hill batch / score / per-variant.
- `evals/rounds/overnight.jsonl` — append-only event log.
- `evals/raw/*.jsonl` — raw inference outputs (per row: id, variant, prompt,
  systemPrompt, response, durationMs).
- `evals/build_round.mjs` — variants spec → batch JSON.
- `evals/score.mjs` — results JSONL → JSON pass-rate report.
- `evals/run_overnight.mjs` — the driver.
- `evals/morning_report.mjs` — summary printer.
